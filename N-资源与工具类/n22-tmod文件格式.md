tModLoader使用自定义格式保存mod文件，文件扩展名为`.tmod`。该文件在tModLoader编译mod时生成，包含mod程序集、资源（如图像或声音）、库以及mod作者选择包含在mod中的任何其他数据。

文件格式相对简单，其布局如下所述。

注意：每个数据元素都使用其对应的`BinaryReader.ReadX`方法编写。如果您正在使用.NET以外的语言编写.tmod文件解析器，请记住字符串的保存方式如下：

| 值 | 数据 | 描述 |
| --- | ---- | ---- |
| 长度 | 4字节 | 这是一个7位编码整数，描述字符串的长度。如果您不熟悉7位编码整数，请搜索 |
| 字符串 | **[长度]**字节 | 实际的UTF-8字符串，由**[长度]**字节组成 |

从这里开始，任何对字符串的提及都指上面的布局。

## .tmod二进制文件结构

| 值 | 数据 | BinaryReader方法 | 描述 |
| --- | ---- | ------------------- | ----------- |
| 文件头 | 4字节 | BinaryReader.ReadBytes(4) | 文件的前4个字节是ASCII字符，拼出'TMOD'。如果文件头无效，则该文件不是.tmod文件或已损坏 |
| 版本 | 字符串 | BinaryReader.ReadString() | mod构建所针对的tModLoader版本 |
| 哈希值 | 20字节 | BinaryReader.ReadBytes(20) | .tmod文件的SHA1计算哈希值 |
| 签名 | 256字节 | BinaryReader.ReadBytes(256) | .tmod文件的签名，用于mod浏览器 |
| 文件数据长度 | 无符号32位整数 | BinaryReader.ReadUInt32() | 读取所有先前数据后文件中剩余的字节数。这些剩余字节用于计算文件前面找到的哈希值 |
| Mod名称 | 字符串 | BinaryReader.ReadString() | 存储在.tmod文件中的mod内部名称 |
| Mod版本 | 字符串 | BinaryReader.ReadString() | 存储在.tmod文件中的mod版本 |
| 文件数量 | 有符号32位整数 | BinaryReader.ReadInt32() | 存储在.tmod文件中的文件数量 |
| **[文件数量]**个文件条目 | 字符串、有符号32位整数、有符号32位整数 | BinaryReader.ReadString(); BinaryReader.ReadInt32(); BinaryReader.ReadInt32() | 字符串是文件的名称，包括其路径。第一个整数是文件未压缩的长度（以字节为单位），第二个整数是文件压缩后的长度。重复读取文件条目**[文件数量]**次以获取所有条目 |
| **[文件数量]**个文件数据 | **[文件条目压缩长度]**字节 | BinaryReader.ReadBytes(**[文件条目压缩长度]**) | 这是所有实际文件数据存储的位置（按照与文件条目相同的相应顺序）。每个文件条目的压缩数据长度表示需要读取的字节数以获取其数据。例如，如果第一个文件条目的压缩长度为34字节，则读取34字节以获取其数据。然后如果第二个的长度为89字节，则读取89字节。重复此操作直到到达.tmod文件的末尾 |

成功解析.tmod文件后，您需要解压缩每个文件的数据，因为tModLoader通常使用DEFLATE算法压缩大多数文件。（如果您在.NET上，一个特别好的解压缩数据的库是Ionic.Zlib）。如果文件条目具有不相等的压缩长度和未压缩长度，则认为该文件已被压缩。
